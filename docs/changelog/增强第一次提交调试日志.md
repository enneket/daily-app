# 增强第一次提交调试日志

## 修复内容
针对用户反馈的"第一次提交还是直接推送"问题，增强了调试日志以便更好地诊断问题。

## 具体修改

### 1. 增强 submitReport() 方法日志
- 在提交前状态检查中添加了 `BATCH_SIZE` 和 `AUTO_PUSH_INTERVAL` 的输出
- 在更新本地缓存时添加了详细的状态跟踪：
  - 显示更新前后的 `pendingCommits` 值
  - 明确标识是否满足推送条件
  - 区分推送成功和保持本地暂存的情况

### 2. 重构 shouldAutoPush() 方法日志
- 添加了详细的条件检查过程：
  - 显示所有相关变量的值和类型
  - 分别检查条件1（累积提交数）和条件2（时间间隔）
  - 对于时间检查，显示具体的时间差和是否超过阈值
  - 当 `lastPushTime` 为空时，明确说明跳过时间检查（首次使用）

### 3. 调试信息格式化
- 使用清晰的分隔符和标题
- 添加 ✅ 和 ❌ 图标表示条件满足情况
- 显示时间差的分钟数便于理解

## 预期效果
通过这些增强的调试日志，可以：
1. 清楚看到第一次提交时的所有状态值
2. 准确判断 `shouldAutoPush()` 的决策过程
3. 快速定位第一次提交直接推送的根本原因
4. 验证本地暂存机制是否正常工作

## 如何查看调试日志
在开发模式下，打开应用的开发者工具（F12），在 Console 标签页中可以看到详细的调试输出。
