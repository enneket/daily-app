# 修复第一次提交立即推送问题

## 问题描述

在 GitHub 提交频率优化功能中，每天的第一次提交仍然会直接提交到 GitHub，而不是按预期暂存到本地。这违背了批量提交和定时提交的设计初衷。

## 根本原因

### 1. 初始化逻辑错误

```typescript
// 问题代码
const lastPushTime = this.store.get('lastPushTime') || Date.now(); // ❌
```

在 `shouldAutoPush()` 方法中，`lastPushTime` 的默认值被设置为 `Date.now()`，导致第一次提交时可能满足"超过 4 小时"的条件。

### 2. 构造函数过度初始化

```typescript
// 问题代码
if (!this.store.get('lastPushTime')) {
  this.store.set('lastPushTime', Date.now()); // ❌
}
```

构造函数中主动设置 `lastPushTime` 为当前时间，进一步加剧了问题。

### 3. 逻辑缺陷

即使修复默认值为 `0`，第一次提交时仍然会因为"距离 1970年超过 4 小时"而触发立即推送。

## 解决方案

### 核心修复：检查 lastPushTime 是否存在

```typescript
// 修复后
private shouldAutoPush(): boolean {
  const pendingCommits = this.store.get('pendingCommits') || 0;
  const lastPushTime = this.store.get('lastPushTime'); // 不设置默认值
  const now = Date.now();
  
  // 策略 1: 累积满 10 个 commit
  if (pendingCommits >= this.BATCH_SIZE) {
    return true;
  }
  
  // 策略 2: 有未提交的内容且距离上次提交超过 4 小时
  // 关键修复：只有在 lastPushTime 存在时才检查时间间隔
  if (pendingCommits > 0 && lastPushTime && (now - lastPushTime) >= this.AUTO_PUSH_INTERVAL) {
    return true;
  }
  
  return false;
}
```

### 关键改进

1. **移除默认值**：`lastPushTime` 不设置默认值，保持 `undefined`
2. **添加存在性检查**：只有在 `lastPushTime` 存在时才检查时间间隔
3. **保持构造函数简洁**：不初始化 `lastPushTime`

## 修复后的逻辑

### 第一次提交场景

```
第一次提交：
  pendingCommits = 1
  lastPushTime = undefined (未设置)
  
检查条件：
  1. pendingCommits >= 10? → false (1 < 10)
  2. pendingCommits > 0 && lastPushTime && 超过4小时? 
     → false (lastPushTime 为 undefined)
  
结果：不推送，保存到本地 ✅
```

### 累积 10 个提交场景

```
第 10 次提交：
  pendingCommits = 10
  lastPushTime = undefined
  
检查条件：
  1. pendingCommits >= 10? → true (10 >= 10)
  
结果：推送到 GitHub，设置 lastPushTime ✅
```

### 定时推送场景

```
第 11 次提交（4小时后）：
  pendingCommits = 1
  lastPushTime = 上次推送时间
  
检查条件：
  1. pendingCommits >= 10? → false (1 < 10)
  2. pendingCommits > 0 && lastPushTime && 超过4小时? 
     → true (1 > 0 && 存在 && true)
  
结果：推送到 GitHub ✅
```
